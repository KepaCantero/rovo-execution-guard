# 🛡️ QUALITY GATES (CRÍTICO)

El sistema debe definir e implementar dos niveles de Quality Gates:

**A. Quality Gates del Producto (Business Logic):**
- **Gate 1 (Definición):** Un ticket no puede pasar a "In Progress" si el Rovo Consistency Score es menor al 80%.
- **Gate 2 (Ejecución):** Un PR en GitHub no puede ser mergeado si el ticket de Jira asociado tiene inconsistencias no resueltas.
- **Gate 3 (Entrega):** Validación cruzada final entre la descripción del PR y el contexto histórico extraído por Rovo.

**B. Quality Gates del Código (CI/CD de la propia App):**
- Cobertura de tests unitarios estricta (Mínimo 85%).
- Zero warnings en ESLint (TypeScript estricto).
- Verificación automática de dependencias y vulnerabilidades (Dependabot/Snyk).
- Linting específico de Forge (`forge lint` automático en pipeline).
- Despliegue bloqueado si los tests de integración fallan.

# 🏗️ SISTEMA DE DEPLOY Y VERSIONADO (NUEVO CORE)

### 1. Gestión de Versiones (SemVer Automatizado)
El sistema debe utilizar **Standard Version** o **Semantic Release** para automatizar el versionado basado en los *Conventional Commits*:
- **Fix:** Genera un `patch` (v1.0.1).
- **Feat:** Genera una `minor` (v1.1.0).
- **BREAKING CHANGE:** Genera una `major` (v2.0.0).
- Cada versión debe generar automáticamente un `CHANGELOG.md` y un *GitHub Release*.

### 2. Ciclo de Vida del Despliegue (Environments)
El pipeline de GitHub Actions debe gestionar tres entornos de Forge:
- **Development:** Despliegue automático tras cada push a `develop`.
- **Staging:** Despliegue tras merge a `main`. Aquí se ejecutan los E2E tests finales.
- **Production:** Despliegue mediante un *Deployment Gate* (aprobación manual) tras generar un Tag de versión.

### 3. Estrategia de Rollback (Reversión Instantánea)
Si las métricas de monitoreo (Sentry/Forge Logs) detectan errores críticos tras un despliegue:
- **Automatización de Rollback:** Workflow de GitHub que ejecuta `forge deploy --environment production --version <stable_version>`.
- **Estado de Versión:** El sistema debe mantener un registro en el repo (`.forge-versions.json`) de la última versión estable probada por los tests E2E.

---

# 🛡️ INFRAESTRUCTURA DE CALIDAD Y OPS
- **Testing:** Unitarios (Jest), Integración, y E2E (Playwright) ejecutados antes de cualquier despliegue en Staging/Prod.
- **Git Hooks:** Husky para `pre-commit` (lint/unit tests) y `commit-msg` (validación de Conventional Commits).
- **GitHub Actions:**
  - `ci.yml`: Validación de PRs.
  - `release-and-deploy.yml`: Automatiza el bump de versión y el deploy a Forge.
  - `rollback.yml`: Ejecución manual/automática para revertir a la versión anterior.

---

# 📊 OBSERVABILIDAD Y MONITOREO
- **Sentry/Datadog:** Captura de errores en tiempo real.
- **Health Checks:** Script post-deploy que verifica que la app responde correctamente en el entorno de producción.

---

# 📦 ENTREGABLES REQUERIDOS

1. **Estrategia de Versionado:** Definición de qué cambios disparan cada nivel de versión en Forge.
2. **Workflows de GitHub Actions:**
   - `.github/workflows/deploy.yml` (con lógica de entornos).
